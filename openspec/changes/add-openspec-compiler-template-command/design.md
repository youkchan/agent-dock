## コンテキスト
`compile-openspec` は柔軟性を持たせつつも、`tasks.md` から task を抽出するために必要な記法がある。
入力者が自由形式で書くと、依存行や対象行が未認識になり、後段の手修正コストが増える。

## 目標
- compile 互換の固定テンプレートをワンコマンドで出力できること。
- 日本語/英語の2種類のテンプレートを提供すること。
- テンプレートは「骨組みのみ」で、内容生成は行わないこと。

## 非目標
- OpenSpec change ディレクトリの自動作成
- proposal/design/spec の自動生成
- タスク内容の推論生成

## 設計判断

### 1) コマンド形
CLI サブコマンドを追加する。
- 例: `python -m team_orchestrator.cli print-openspec-template --lang ja`

デフォルト出力先は stdout とし、利用者がそのまま貼り付けて使える形にする。

### 2) 言語切替
`--lang` を `ja|en` で受ける。
- `ja`: 見出し/補助文を日本語で出力
- `en`: 見出し/補助文を英語で出力

どちらの言語でも、`compile-openspec` が解釈できるキーを使う。

### 3) 固定テンプレート内容
テンプレートは次の必須構造を含む:
- 実装タスクセクション
- `- [ ] <task-id> <title>` 行
- 依存行（`依存` または `depends_on`）
- 対象行（`対象` または `target_paths`）
- 成果物/説明行（`成果物` または `description`）
- 検証セクション（`検証項目` / `Verification Checklist`）
- チェックリスト行

### 4) 安全性
テンプレート生成は純粋出力で副作用を持たない。
無効な `--lang` は CLI エラーにする。

## リスクと緩和
- リスク: parser 更新とテンプレート乖離
  - 緩和: テストでテンプレート文字列を compile パーサに通し、最低1件の task 抽出を検証する
- リスク: 利用者がプレースホルダを残してしまう
  - 緩和: プレースホルダは明示トークン（`<...>`）にして注意喚起する
